Sprint 15 Backlog: GitLab CI/CD Pipeline
Setup
Epic: As a Development Team, we want to automate the build, test, and containerization process of our LibraryWeb project using GitLab CI/CD, so that we can guarantee the main branch is always stable and a ready-to-deploy Docker image is automatically generated.
User Stories & Acceptance Criteria
User Story 1: Initialize the Pipeline & Stages
As a developer, I want to establish the basic structure of our CI/CD pipeline so that jobs run in the correct chronological order.
• [ ] Create a file named exactly .gitlab-ci.yml in the root directory of your repository.
• [ ] Define the four core stages at the top of the file: compile, test, package, and dockerize.
• [ ] Commit and push the file. Verify in the GitLab UI (under Build -> Pipelines) that a pipeline was triggered (even if it does nothing yet).
User Story 2: Automate Maven Build & JUnit Tests
As a QA engineer, I want the pipeline to compile our Java code and run all unit tests in a clean environment.
• [ ] Create a job named build-java in the compile stage using the maven:3.9.4- eclipse-temurin-17 image that runs mvn clean compile.
• [ ] Create a job named test-java in the test stage using the same image that runs mvn test.
• [ ] Create a job named package-app in the package stage that runs mvn package.
• [ ] Crucial: In the package-app job, define an artifacts block to save the generated target/*.war file so it can be passed to the next stage. Set it to expire in 1 hour.
User Story 3: Build & Push to GitLab Container Registry
As a DevOps engineer, I want the pipeline to build our Docker container and push it securely to our private GitLab Registry.
• [ ] Create a job named build-and-push-docker in the dockerize stage.
• [ ] Use the docker:24.0.5 image and the docker:24.0.5-dind service (Docker-in- Docker).
• [ ] Write a script that logs into the registry using the predefined variables: docker login $CI_REGISTRY -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD.

• [ ] Build the image and tag it properly: docker build -t
$CI_REGISTRY/$CI_PROJECT_PATH/libraryweb:latest .
• [ ] Push the image to the registry: docker push
$CI_REGISTRY/$CI_PROJECT_PATH/libraryweb:latest.
• [ ] Add a rules block so this job only runs if the commit is on the main branch.
User Story 4: Test the Safety Net (Break the Build)
As a Team Lead, I want to verify that the pipeline actually catches bad code before it merges into main.
• [ ] Create a new feature branch (e.g., feature/break-test).
• [ ] Intentionally change a JUnit test so that it fails (e.g., assertEquals(1, 2);).
• [ ] Push the branch and open a Merge Request (MR) in GitLab.
• [ ] Verify that the pipeline fails at the test stage and GitLab shows a red "Pipeline failed" warning, blocking the merge.
• [ ] Fix the test, push again, and verify the pipeline turns green.
Starter Code Snippets
1. The .gitlab-ci.yml Blueprint
YAML
# Define the chronological order of execution
stages:
- compile
- test
- package
- dockerize # Job 1: Compile the code
build-java:
stage: compile image: maven:3.9.4-eclipse-temurin-17
script:
- echo "Compiling the code..."
- mvn clean compile # Job 2: Run the tests
test-java:
stage: test image: maven:3.9.4-eclipse-temurin-17
script:
- echo "Running JUnit tests..."
- mvn test # Job 3: Create the .war file and save it as an Artifact
package-app:
stage: package image: maven:3.9.4-eclipse-temurin-17
script:

- echo "Packaging the application..."
- mvn package
artifacts:
paths:
- target/*.war expire_in: 1 hour # Don't waste server storage! # Job 4: Build Docker Image and push to GitLab Registry
build-and-push-docker:
stage: dockerize image: docker:24.0.5
services:
- docker:24.0.5-dind
rules:
- if: $CI_COMMIT_BRANCH == "main" # Only run on main branch!
script:
- echo "Logging into GitLab Container Registry..."
- echo "$CI_REGISTRY_PASSWORD" | docker login $CI_REGISTRY -u $CI_REGISTRY_USER --password-stdin
- echo "Building the Docker Image..."
- IMAGE_TAG=$CI_REGISTRY/$CI_PROJECT_PATH/libraryweb:latest
- docker build -t $IMAGE_TAG .
- echo "Pushing the Image..."
- docker push $IMAGE_TAG
Definition of Done (DoD)
Demonstrate the following to the Agile Coach:
1. Open your project in the GitLab web interface.
2. Navigate to Build -> Pipelines and show a completely green pipeline for your main branch.
3. Click on the package-app job and demonstrate that you can download the .war artifact directly from the web browser.
4. Navigate to Deploy -> Container Registry and show that your libraryweb:latest Docker Image has been successfully uploaded by the CI/CD pipeline.
