Short description: You will implement a desktop application for managing student theses. The application uses Java Swing as the UI (inbound adapter) and an existing DAO/JDBC persistence layer(outboundadapter), withH2asthedefaultdatabase. InadditiontoCRUD,youwillimplement workflow actions Submit, Approve, Reject. The architecture follows Hexagonal Architecture (Ports & Adapters) and the work is organized using Scrum.
1. Learning objectives
After completing Phase 1, you can:
• structure a Swing/JDBC project using hexagonal architecture (ports/adapters), • apply an MVP/MVC-like separation in Swing (view, presenter, use cases), • enforce workflow rules in the core (not only in the UI), • keep the UI responsive by running DB operations off the EDT (SwingWorker), • integrate existing DAO implementations properly as outbound adapters.
2. Constraints & prerequisites
2.1 Provided
• A DAO already exists. Do not rebuild the DAO.
• Database: H2 (default). Optional: MySQL via configuration.
• Technologies: Java (recommended: 17+), Swing, JDBC.
2.2 Time & process
• Work is carried out as a Scrum sprint (e.g., one week / until the next lab session).
• Minimal Scrum events: Planning, short dailies during lab, Review (demo), Retrospective.

4. Domain model & workflow
4.1 Entity Thesis
Required fields (Phase 1):
• id (unique, managed by DB/DAO)
• title (must not be empty)
• studentId (must not be empty)
• supervisorId (optional in Phase 1)
• status (see below)
• rejectReason (only for REJECTED, must not be empty)
4.2 Status (enum)
• DRAFT
• SUBMITTED
• APPROVED
• REJECTED
4.3 Workflow rules (must be enforced in the core)
• Submit: only from DRAFT to SUBMITTED
• Approve: only from SUBMITTED to APPROVED
• Reject: only from SUBMITTED to REJECTED and rejectReason must not be empty
Note (State Pattern): In Phase 1, the State Pattern is not required. A correct solution using enum ThesisStatus and guarded transitions is fully acceptable. Optional (stretch): Refactor to the State Pattern only if UI and use-case APIs remain stable.
5. Architecture requirements (Hexagonal Architecture)
5.1 Strict boundaries
• The core must not depend on Swing or JDBC/DAO.
• Swing is an inbound adapter and calls inbound ports (use cases).
• DAO/JDBC is an outbound adapter and implements outbound ports.
• No DAO access from UI classes. The UI talks only to use cases/ports.
5.2 Recommended package structure (example) de.yourorg.thesis core domain

7. Product backlog (user stories & acceptance criteria)
EPIC A: CRUD & listing
US-A1: List theses
As a user, I want to see all theses in a list so that I can manage them.
AC:
• Table shows at least: ID, Title, StudentId, SupervisorId, Status
• Refresh does not block the UI
• Errors are shown clearly
US-A2: Create thesis (DRAFT)
As a user, I want to create a new thesis so that it is recorded in the system.
AC:
• Title and StudentId are required
• Initial status is DRAFT
• After Create: list is refreshed
US-A3: Edit thesis
As a user, I want to edit a thesis so that I can correct data.
AC:
• Only possible when a thesis is selected
• Title must not be empty
• Persistence via DAO/repository adapter
US-A4: Delete thesis
As a user, I want to delete a thesis so that incorrect entries are removed.
AC:
• Confirmation dialog before deletion
• After Delete: list is refreshed and selection is cleared
EPIC B: Workflow
US-B1: Submit
As a user, I want to submit a thesis so that it can be reviewed.
AC:
• Only DRAFT can become SUBMITTED
• Status change is persisted
• Illegal transitions return a clear error message
US-B2: Approve
As a user, I want to approve a submitted thesis so that it becomes final.
AC:
• Only SUBMITTED can become APPROVED
• Status change is persisted
US-B3: Reject (with reason)
As a user, I want to reject a submitted thesis and provide a reason to document the decision.
AC:
• Only SUBMITTED can become REJECTED
