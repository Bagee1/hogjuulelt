Introduction to GitLab CI/CD .

Combines Source Code Management (Git), Issue Tracking, CI/CD pipelines, and Container Registry all in a single application ecosystem.

ICSI486 Software Construction Slide 7

Architecture Infrastructure
GitLab Architecture: Server vs. Runners
Key Concepts
Separation of Concerns
Component Roles
The GitLab Server does not execute code itself. This prevents heavy build processes (compiling Java, building Docker images) from slowing down the web interface.
GitLab Server
The "Brain". Hosts the repository, provides the web interface, manages users, and stores pipeline configurations. It triggers pipelines based on events (like a push).
Runner Types
Triggers Job Shared Runners: Provided by GitLab.com (free quota).
Specific Runners: Hosted on your own university or company servers for security, control, and unlimited performance.
GitLab Runners
The "Muscle". Separate worker machines (servers, VMs, or containers) that pull jobs and execute the actual scripts defined in your .gitlab-ci.yml.
Execution Environment
Runners typically use the Docker executor , meaning every job runs in a fresh, isolated container environment, ensuring no leftovers from previous builds.

Configuration
The Anatomy of a Pipeline .
Understanding the .gitlab-ci.yml structure
Strict YAML Syntax
Written in YAML (YAML Ain't Markup Language). Indentation using spaces (not tabs) is strictly enforced to define structure.
Pipeline Composition
The top -level structure. Pipelines are composed of distinct stages that organize the workflow logic.
Stages & Jobs stages act as categories. jobs are the specific tasks assigned to those stages.
Scripts
Jobs contain the actual script section —the shell commands (e.g., mvn test) the Runner will execute.
Execution Flow
Jobs in the same stage run in parallel . Jobs in subsequent stages run sequentially (wait for previous to finish).

Infrastructure
The GitLab Container Registry .
A Private Home for Your Images
GitLab provides a built -in Docker Registry for every project. There is no need to push proprietary code to the public Docker Hub.
Registry URL Variable
$CI_REGISTRY automatically points to the correct registry address for your specific GitLab instance.
Auto -Authentication
$CI_REGISTRY_USER & $CI_REGISTRY_PASSWORD are auto -generated credentials, valid only for the duration of the job (Zero Trust).
Namespace & Path
$CI_PROJECT_PATH defines your specific namespace (e.g., user/libraryweb ) to ensure images are tagged correctly.

Security Best Practices
Protecting Secrets with CI/CD Variables .
The Golden Rule
Never commit passwords, SSH keys, or API tokens directly to .gitlab-ci.yml.
Secure Configuration
Navigate to Settings → CI/CD → Variables in your GitLab project to add sensitive credentials like PRODUCTION_SSH_KEY.
Log Protection
Always check the "Mask variable" option. This ensures secrets are redacted (replaced with [MASKED]) if they accidentally appear in job logs.
Access in Pipeline
GitLab injects these secrets as standard environment variables. Access them in your YAML scripts via $PRODUCTION_SSH_KEY.

Troubleshooting
The Pipeline is RED . Now what-
1
Don't Panic
A failing pipeline is a good thing ! It means the system successfully caught a bug before it reached production. Treat it as a safety net, not a punishment.
2
Inspect the Job Logs
Click on the failed job in the GitLab UI. Read the terminal output carefully. The actual error message (e.g., Compilation Failure or AssertionError) is usually found at the very bottom of the log.
3
Fix Locally, Push Again
Reproduce the issue on your local machine. Fix the underlying Java code or the broken test. Do not try to hack the .gitlab-ci.yml just to force it to pass.
