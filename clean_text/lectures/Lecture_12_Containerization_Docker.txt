Modern Architecture
Containers (The Modern Way)
OS-Level Virtualization
Run on the Docker Engine
Container Architecture
Replaces the heavy Hypervisor. The engine manages containers efficiently.
App A App B App C
Share the Host OS Kernel
Docker Engine
No duplicate Operating Systems. All containers leverage the same Linux kernel.
Host OS
No Guest OS required
Infrastructure
Only the App and its exact dependencies (bins/libs) are packaged inside.
Lightweight & Efficient
Start time: ~50ms Size: ~MBs (not GBs)
Boots in milliseconds, occupies minimal disk space, and maximizes hardware usage.

Port Mapping Explained
Connecting Containers to the World
Network Isolation Practical Examples
Theory & Concepts Usage Scenarios
Invisible by Default Standard Mapping
-p 8080:8080
Containers are completely isolated. A service running on port 8080 inside is invisible to your laptop.
Access via http://localhost:8080. Direct 1-to-1 mapping.
Punching a Hole
Port Redirection
You must explicitly open a connection through the container's "firewall" to -p 9000:8080 allow external traffic in.
Access via http://localhost:9000. Useful if port 8080 is already busy on your laptop.
The Syntax
The logic is always: Host Port : Container Port. (Left side = Your No Mapping-
Laptop, Right side = Inside Container).

Docker Volumes
Saving data beyond the container's lifecycle
The Volume Flag (-v) bash
-v host_path:container_path
1 # Mount host folder to container This creates a direct link ("mount") between your laptop's filesystem and the container. The container thinks it's writing to its own internal folder, but it's actually writing to your hard drive.
2 docker run -d \
3 --name library-db \
4 -v/my/local/data:/var/lib/mysql \
Data Survival

Persistence

Without volumes, if you delete a container, its data is gone forever.
With volumes, the data survives! This is critical for databasesand user uploads.
