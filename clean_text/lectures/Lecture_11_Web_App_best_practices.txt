Security Risk
The Over - Posting Problem
Why DTOs for Input-
Mass Assignment Vulnerability
Hackers can inject fields like isAdmin: true into
JSON payloads using tools like Postman, even if your
HTML form doesn't show them.
The Direct Mapping Flaw
If you map JSON directly to your database entity
(User), frameworks often blindly copy all matching fields, overwriting sensitive data.
The DTO Solution
Use a specific UserCreateDTO that only contains safe fields. Any extra data sent by the hacker is simply ignored.

Best Practice 3
Input Validation
Trust No One
Never trust user input. Treat all external input as potentially malicious and tampered with.
Client -Side (HTML5)
Attributes such as <input type="email" required> improve the user experience but offer no security whatsoever. They can easily be bypassed in the browser.
Server -Side (Java)
Validation must take place within the servlet before data reaches the business logic or database.

Validation Logic
Serverseitige Validierung: Fail Fast
The Principle
"Fail Fast" means verifying preconditions immediately upon entry. If inputs are invalid, stop execution instantly.
Prevents bad data from reaching the Core
Saves processing resources
Avoids deep nesting (Arrow Code)
HTTP 400 Bad Request
Standard response for client errors. Never return 200 OK with an error message in the body!

Performance
Pagination
Chunks vs. Everything
Performance Impact
Fetching 1 million rows causes Out of Memory errors. Fetching 20 rows is instant.
API Contract
Translates to SQL
The client requests a specific page index and size (e.g., page 1, 20 items) via URL parameters.
Database Efficiency
Use SQL LIMIT and OFFSET to skip rows at the database engine level, saving memory.
