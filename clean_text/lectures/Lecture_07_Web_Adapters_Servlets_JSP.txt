To write Web Adapters in Java, we need access to the Jakarta EE API standard libraries.
Jakarta Servlet API
This library provides the essential interfaces like HttpServlet, HttpServletRequest, and annotations like @WebServlet.
Dependency Scope: "Provided"
This is crucial! We tell Maven that we need these classes to compile our code, but we do
NOT want them packaged in our final .warfile.
Why "provided"-
Apache Tomcat acts as the container and already includesimplementation libraries for the Compile Time: OK Runtime (Tomcat): OK Servlet API. If we include them again, we risk class loading conflicts and runtime errors.

BASIC SERVLET STRUCTURE
WHAT IS A SERVLET-
Definition:
A Servlet is a Java class that extends the capabilities of a server to handle
HTTP requests and generate dynamic responses.
Architectural Role:
In our Web Adapter architecture, the Servlet acts as the CONTROLLER.
It receives the request, processes logic (via Service), and determines the view.
Key Components:
Inherit base HTTP handling capabilities.
Define URL routing (e.g., "/theses").
Implement doGet() or doPost() logic.

WRITING THE JSP VIEW
The JSP file theses.jsp is where we convert our Java objects into HTML. /WEB-INF/views/theses.jsp HTML + Java Here's how the unpacking process works:
1 Page Imports
Just like in Java classes, we must import any types we use (List, ThesisDTO) using the
@page importdirective at the top.
2 Unpacking the "Backpack"
We use request.getAttribute("key")to retrieve data sent by the Servlet. Note that this returns an Object, so we must castit back to List<ThesisDTO>.
3 Java Iteration
We can write standard Java loops inside scriptlet tags <% ... %>to generate repetitive
HTML rows.
4 Expression Output
The special syntax <%= variable %>prints a value directly into the HTML output stream
(replacing System.out.println).
