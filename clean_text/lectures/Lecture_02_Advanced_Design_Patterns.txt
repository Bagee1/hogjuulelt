The "Anti-Pattern"
When to Avoid Singletons
Critical Drawbacks The Modern Alternative
Testing Nightmare Dependency Injection (DI)
Singletons introduce hidden dependenciesinside methods. Mocking Enterprise Standard (Spring, Guice) `static` methods is difficult (requires Powermock) and leads to brittle tests. Instead of asking for dependencies ("Give me the Logger"), classes declare them ("I need a Logger"). The framework handles the lifecycle (Singleton Scope).
Global State
State persists between unit tests. Test A modifies the Singleton, causing Test B to fail mysteriously. Violates test isolation.
SRP Violation
The class manages its own lifecycle ANDits business logic. It does too much.

Factory Implementation
Programming to an Interface
UserFactory.java
1 The Interface
The client relies only on ThesisUser. It doesn't know (and doesn't care) that the actual object is a Student. This is Loose Coupling.
2 Centralized Creation
Complex logic like ID generation, validation, or dependency injection happens here, not in the client code.
3 Clean Client Code
The client simply requests a type via create(). No newkeyword in the main application logic.
Avoid "Stringly-Typed" Code:
Don't pass strings like "student" to your factory. Use an enum
UserTypeto catch typos at compile time!

Common Pitfalls
Singleton & Factory Mistakes to Avoid
PitfallsComparison.java
The volatile Trap
Without volatile, the JVM might reorder instructions. Thread B could see a non-null instthat points to memory not yet fully initialized by the constructor.
Serialization Breaks Singletons
Deserializing a Singleton creates a new instance! Fix:Implement readResolve()to return the existing instance.
Factory: "Stringly-Typed"
Avoid create("student"). Typos cause runtime crashes. Fix:Use Enums
(UserType.STUDENT) for compile-time safety.
Best Practice:
Use a single-element enum for Singletons if you don't need lazy loading or inheritance. It handles serialization and reflection attacks automatically!
